# 01 · The Struct Skeleton (Python)

Python port of [`week1_baseline/ruby/01_struct_skeleton`](../../ruby/01_struct_skeleton/README.md).
It adds the lightweight data structures passed through later Boukensha steps:
`Tool`, `Message`, and `Context`.

## Environment setup

This repo uses one shared virtual environment at the repository root. Create it
once, then install this step's dependencies:

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -r week1_baseline/python/01_struct_skeleton/requirements.txt
```

## Data structures

### `Tool`

Describes a callable tool available to the agent.

| Field | Description |
|---|---|
| `name` | The tool name |
| `description` | Shown to the agent so it knows when to invoke the tool |
| `parameters` | A dictionary of argument names to schemas |
| `block` | The callable run when the tool is invoked |

```python
Tool(
    "move",
    "Move the player in a direction",
    {"direction": {"type": "string"}},
    lambda direction: f"You move {direction}.",
)
```

String form:

```text
#<Tool name=move description=Move the player in a direction params=['direction']>
```

### `Message`

A single unit of conversation between the user and the agent.

| Field | Description |
|---|---|
| `role` | Who is speaking: `user`, `assistant`, or `tool_result` |
| `content` | Text generated by the agent or supplied by the user |
| `tool_use_id` | Optional ID linking a tool result to its tool call |

```python
Message("user", "Explore north and tell me what you find.")
Message("tool_result", "You move north.", "toolu_01X")
```

String forms:

```text
#<Message role=user content=Explore north and tell me what you find....>
#<Message role=tool_result [toolu_01X] content=You move north....>
```

### `Context`

Holds the task, system prompt, conversation history, and registered tools needed
for an API call.

| Field | Description |
|---|---|
| `task` | The task class, such as `Player` |
| `system` | The system prompt, sent separately from messages |
| `messages` | The full conversation history |
| `tools` | Registered tools keyed by name |

```python
ctx = Context(task=Player, system=system_prompt)
ctx.register_tool(tool)
ctx.add_message("user", "Explore north.")
```

String form:

```text
#<Context task=player turns=1 tools=1>
```

## Configuration compatibility

The Python and Ruby implementations read the same `.boukensha/` directory.
Resolution order is:

1. `BOUKENSHA_DIR`
2. `~/.boukensha`

The shared directory supports the same `settings.yaml`, `.env`, and
`prompts/<task>/system.md` override layout.

## Files

| File | Purpose |
|---|---|
| `boukensha/config.py` | Shared configuration loader |
| `boukensha/tool.py` | `Tool` dataclass |
| `boukensha/message.py` | `Message` dataclass |
| `boukensha/context.py` | `Context` container |
| `boukensha/tasks/` | Stateless task classes |
| `examples/example.py` | Runnable smoke test |

## Run example

```bash
./week1_baseline/bin/python/01_struct_skeleton
```

Expected output, with the config directory varying by machine:

```text
=== Boukensha Step 1: Struct Skeleton ===

Config:   #<Boukensha::Config dir=/path/to/repo/.boukensha tasks=player>
Context:  #<Context task=player turns=2 tools=1>
Tool:     #<Tool name=move description=Move the player in a direction (north, so params=['direction']>
Messages:
  #<Message role=user content=Explore north and tell me what you find....>
  #<Message role=assistant content=Sure, let me head north and take a look....>
```

Python uses string dictionary keys, so tool parameters display as
`['direction']` rather than Ruby's `[:direction]`.
